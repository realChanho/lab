AI는 코드를 못 짜는 게 아니다. 너무 잘, 너무 빨리, 너무 자신 있게 짜는 게 문제다. 그래서 브레이크가 필요하다. 안드레이 카파시가 트위터에 올린 글을 누군가 65줄짜리 CLAUDE.md로 정리했고 GitHub 스타 10만을 받았다.

진단

AI 코딩의 3대 고질병

왜 브레이크가 필요한가.

1. 잘못된 가정. "로그인 추가해줘"에 묻지도 않고 JWT 200줄을 우르르 짠다. 정작 원한 건 세션 쿠키 한 줄일 수도.
2. 코드 부풀리기. 3줄이면 될 이메일 검증을 옵션 8개·설정파일·추상 인터페이스까지 붙여 200줄로. "나중에 필요할 수도" → 그 나중은 99% 안 온다.
3. 멋대로 고치기. 함수 하나 고쳐달랬더니 옆 함수 포매팅·변수명·주석까지 손대 리뷰할 변경이 50개로 불어난다.
처방

이를 막는 4원칙

65줄 CLAUDE.md의 뼈대.

  1. 코딩하기 전에 먼저 생각하라
    가정하지 마. 헷갈리면 숨기지 말고 트레이드오프를 알려줘.
    • 지금 어떻게 이해했는지 명확히 말하고, 헷갈리면 묻기
    • 해석이 여러 개면 다 보여주고 혼자 고르지 않기
    • 더 단순한 방법이 있으면 사용자에게 반박이라도 하기
    • 모르는 게 있으면 멈추고 묻기 — 고질병 1 차단
  2. 단순하게 먼저
    문제를 해결하는 최소 코드. 추측은 없이.
    • 시키지 않은 기능·유연성·설정 가능성 넣지 않기
    • 한 번만 쓸 코드를 추상화하지 않기
    • 일어날 수 없는 상황에 에러 처리 넣지 않기
    • 200줄이 50줄로 가능하면 다시 짜기 — 고질병 2 차단
  3. 외과 수술처럼 정확하게
    꼭 수정해야 하는 것만 고쳐라.
    • 인접 코드·주석·포매팅 개선 금지, 시키지 않은 리팩토링 금지
    • 기존 스타일에 맞추기 (내가 다르게 짤 거라도)
    • 죽은 코드는 말만, 지우지 않기
    • 모든 변경된 줄은 사용자 요청으로 추적 가능해야 — 고질병 3 차단
  4. 목표 중심으로 실행
    성공 기준을 정해 검증될 때까지 반복해.
    • "검증 추가해줘" 대신 "예상 못한 입력 테스트 작성 → 통과시키기"
    • "버그 수정" 대신 "버그 재현 테스트 먼저 → 통과되게"
    • 여러 단계면 짧은 계획(단계·검증)을 먼저 적게 하기
    • 강한 성공 기준 = AI가 혼자 돌아감 / 약하면 매번 확인 필요
