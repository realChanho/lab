프로젝트 문서 구조 기준

CLAUDE.md vs 스펙/컨벤션 — 새 프로젝트 시작 시 잡을 표준 구조 · 정리일 2026-05-17

핵심 원칙: "항상 로드돼야 하는 것"과 "필요할 때만 참조하는 것"을 분리한다. CLAUDE.md는 매 세션 토큰을 먹으므로 짧게, 상세 내용은 별책으로 뺀다.

4계층 구조

1

CLAUDE.md (프로젝트 루트) 자동 로드

매 세션 컨텍스트에 자동 포함. 짧고 늘 지켜야 할 것만 넣는다.

  • 행동 규칙 (카파시 4원칙 + 추가 규칙)
  • 빌드/테스트/실행 명령어 — npm test, pytest 등
  • 핵심 컨벤션 요약 (몇 줄): "함수형 컴포넌트만", "에러는 Result 타입으로"
  • 디렉터리 구조 한눈 요약

길어지면 토큰 낭비 → 아래 계층으로 분리.

2

상세 스펙 / 컨벤션 필요 시 참조

자동 로드 안 됨. 두 방법으로 연결한다.

  • @import — CLAUDE.md 안에 @docs/conventions.md 한 줄 → 그 파일도 자동 로드. 항상 필요한 컨벤션이면 이걸로.
  • 언급만 — "상세는 docs/conventions.md 참고"라고만 적고 필요할 때 읽게. 가끔만 필요하면 토큰 효율적.
3

기능 스펙 필요 시 참조

기능 단위 명세. docs/spec/auth.md 처럼 분리하고 작업 시 해당 파일만 읽게 한다.

4

진행 중 작업 상태 작업 중 갱신

CLAUDE.md 7번 규칙(Plan + Checklist + Context Notes)이 규정하는 산출물.

  • checklist.md — 할 일 체크박스, 진행하며 체크
  • context-notes.md — 작업 중 의사결정과 근거, 계속 append

권장 디렉터리 레이아웃

project-root/
├── CLAUDE.md            # 행동 규칙 + 핵심 명령어 (자동 로드, 짧게)
├── checklist.md         # 진행 중 작업 체크박스
├── context-notes.md     # 의사결정 로그
└── docs/
    ├── conventions.md    # 코딩 컨벤션 상세 (네이밍, 폴더링, 스타일)
    ├── architecture.md   # 시스템 구조, 의사결정 기록
    └── spec/
        └── auth.md        # 기능별 스펙
